# 유인–3자 콤비네이션 설계 (Bait → Third-Man → Drop-Receive)

게임의 시그니처가 되어야 할 "유인 축구"를 정직한 기하로 다시 세운다.
사용자 지적에서 출발: **유인과 탈출은 두 동작이 아니라 하나의 콤비네이션이다.**

## 1. 문제 — 지금 모델은 기하가 부정직하다

현재: 캐리로 유인 → 압박수 커밋(full_commit) → 커밋수비수 **뒤쪽**에 윈도우가
열림 → pass_space로 그리로 패스.

- 윈도우가 "끌려온 수비수 뒤"에 열리는데(코드도 그렇게 둔다), 그 수비수가
  볼과 윈도우 사이에 있다. **캐리어는 자기가 연 공간을 직접 쓸 수 없다** — 앞이
  막혔으니까. evaluateLane이 이 직접 패스를 고위험으로 값 매기는 건 맞지만,
  화면은 "쓸 수 없는 공간"을 열린 것처럼 보여준다 → 반쯤 부정직한 시그널.
- 유인(캐리)과 탈출(3자)이 **커플링되지 않음**. 3자는 우연히 거기 서 있으면
  쓰이는 정적 존재. "당기는 동안 저 선수가 내려와 받는다"는 결합이 없다.
- 그래서 캐리 유인은 goal +1%짜리 소소한 스텝일 뿐, 시그니처 무브가 못 됨.
  hold 부양은 애초에 오답 — **유인의 원시동작은 hold(수동)가 아니라 마커를
  향한 캐리(능동적 도발)** 다.

## 2. 올바른 모델 — 하나의 콤비네이션, 세 역할

한 콤비네이션이 세 역할을 **동시에** 요구한다:

1. **BAITER(당기는 나)**: 특정 마커를 향해 캐리한다. **임계 거리** — 마커가
   튀어나와도 나에게 닿기 직전까지. 너무 멀면 안 끌려오고, 너무 가까우면 뺏긴다.
2. **마커 커밋**: 끌려나온다. 뒷공간이 열린다. **단, 베이터의 앞은 이제 막혔다**
   (마커가 볼과 공간 사이). → 직접 패스 불가.
3. **RELEASER(3자/측면 지원)**: 막힌 베이터의 탈출 각을 제공한다 — 레이오프
   또는 옆으로 열어준 레인.
4. **RECEIVER(마킹당하던 우리 선수)**: 정적으로 서 있지 않는다. **압박(떠난
   마커)을 따라 뒷공간으로 내려오며**, RELEASER를 거친 볼을 **전진 방향으로**
   받는다. 내려오며 받았기에 **앞이 크게 열려 있다** → 패스 성공이 아니라
   **라인 브레이크(전진)**.

가치의 원천은 4번(리시버의 오리엔테이션 + 앞 공간)이고, 그것을 1–3이 잠금 해제한다.

## 3. 엔진 매핑 — 새 시스템 아님, 기존 문법 커플링

엔진에 이미 있는 것:
- **오리엔테이션 문법**(space.js:191 computeOrientation): FACING(앞 봄, 전액션)
  / HALF(반몸, 전진 패스 페널티) / BACK(마킹당해 등짐, 전진 거의 불가).
  moving=true면 FACING. → "내려오며 받아 앞을 본다"가 곧 FACING 리셉션.
- **트리거**(press.js:24): carry 1.10, third_man 1.00, bounce 1.05. → RELEASER
  경유 = third_man/bounce.
- **linesBroken**(전진 가치), **superiority**(수적 우위 = 3자가 거기 있음의 조건).

즉 콤비네이션은 이 넷을 하나로 묶는 것:
`캐리(유인) → 마커 커밋(윈도우) → third_man 릴리스 → 리시버 FACING 리셉션 in 뒷공간 = linesBroken`.

**이미 있는 토대 — 카운터-드롭 포크(press.js:191-218, §drop_lab)**:
- `d.markId`: 각 수비수의 담당 마크(우리 선수). 결정 #1의 마커→담당선수 훅.
- 주석 그대로: *"man-marker's target drops deep → follow: marker vacates his
  original slot (window for a third runner)"*. **마킹당한 선수가 깊이 내려가면
  마커가 따라가 자기 슬롯을 비우고, 그 빈 슬롯이 제3의 러너를 위한 창**이 된다.
  우리 콤비의 핵심(리시버 드롭 → 마커 추종 → 3자 공간)이 부분 구현돼 있음.
- 지금은 이게 **상대 AI 내부 포크**(markDropped 감지 → follow/passoff)일 뿐,
  플레이어가 캐리로 **능동 유발**하고 3자 릴리스로 **완성**하는 결합이 없다.
  설계 = 이 기존 메커니즘을 플레이어 트리거·정직 시각화와 커플링하는 것. 저위험.

## 4. 플레이어 표현 — 턴제 결정으로 어떻게 치나

이 게임은 클릭 결정 게임이다. 콤비네이션을 2액션 시퀀스로 편다:

**액션 1 — 유인 캐리(bait carry)**
- 마커를 향해 캐리하되 **도발 밴드**(마커로부터 2~4m)에서 멈추면 유인 성립.
- 그 순간: (a) 마커 커밋 롤(유인 성공 확률은 거리로 — 가까울수록↑ 뺏길 위험도↑),
  (b) 성공 시 **RECEIVER가 뒷공간으로 드롭 시작**(armed), (c) 직접 패스 레인은
  마커가 막은 걸로 표시(적색).

**액션 2 — 릴리스(release)**
- 드롭한 리시버로 가는 길은 **3자 경유(레이오프/측면 지원)로만** 열린다.
- 이 릴리스가 성공하면 리시버는 FACING으로 뒷공간에서 받아 라인 브레이크.
- 직접 패스를 고르면? 막힌 레인 → 고위험(가로채기). "쓸 수 없는 공간"을
  화면이 정직하게 말한다.

→ 결국 **읽기 게임**: 유인이 걸리면 "내 앞은 막혔고, 열린 건 내려온 3자의 길"이
피치에 그려지고(2R 인텔 화살표처럼, 이번엔 우리 콤비네이션으로), 플레이어는
직접 패스의 함정 대신 3자 릴리스를 읽어야 한다.

## 5. 임계 거리 — 스킬 축 (신규)

캐리 목적지와 마커 사이 거리 `d`:
- `d > FAR`(예: 6m+): 마커 커밋 안 함. 유인 실패(공만 앞으로).
- `PROVOKE_BAND`(예: 2~4m): 유인 성립. 커밋 확률 = 거리 반비례.
- `d < NEAR`(예: <2m): 뺏길 위험 급증(기존 carry 경로 태클 리스크 재사용).

→ "닿기 직전"이 스위트스폿. 기존 carry의 경로 태클 리스크(engine.js planCarry)를
그대로 쓰고, 도발 강도만 거리로 추가한다. 플레이어가 통제하는 리스크/리워드.

## 6. 정직한 기하 — 핵심 불변식 (A안 확정, 2026-07-04)

**설계 갈림길**: 엔진 evaluateLane은 "패서 3m 이내 수비수는 레인을 막는 게 아니라
볼을 압박할 뿐 — 릴리스가 그를 지나쳐 나간다"(ISSUE-007)로 모델링돼 있다. 유인은
근접(2~3m)이라 끌려온 마커가 캐리어 코앞 → **직접 패스가 "레인 차단"으로는 안 막힘.**
→ **A안 채택**: 가치를 레인 차단이 아니라 **오리엔테이션+전진**으로.

- **가치의 원천 = 리시버가 내려오며 FACING(전진 방향)으로 받는 것.** 직접 패스는
  리시버를 제자리에 정적으로 두니 전진(라인 브레이크)이 없다. 릴리스는 리시버를
  뒷공간으로 드롭시켜 FACING + 라인 브레이크.
- "내 앞이 막힌다"의 진짜 의미 = "직접 패스로는 전진이 안 된다(정적·마킹 잔재)".
  릴리스가 그 탈출. 엔진의 기존 FACING/BACK 문법을 정직하게 사용.
- **리시버의 움직임이 오리엔테이션을 결정한다**: 내려오며 받으면 FACING(전진),
  서서 받으면 BACK/정적. 이 차이가 콤비네이션의 가치 전부.
- **라인 브레이크 = 볼의 실제 전진(캐리어→드롭)**: 커밋한 마커 라인을 넘어
  뒷공간으로 들어간 만큼. 릴리서는 옆 릴레이라 무관.
- Phase 1 실측(현실적 위치): 릴리스 성공 97% · **FACING 100% · 라인브레이크 100%**.

## 7. 측정 계획 (자기대국)

`scripts/probe-bait-combo.mjs`:
- 직접 패스 시도 시 실패율(가로채기)이 3자 릴리스보다 유의하게 높은가 (막힘 검증).
- 3자 릴리스 성공 시 리시버 FACING 비율·linesBroken 발생률 (전진 가치 검증).
- 유인 캐리 임계거리별 커밋 성공률/태클 위험 곡선 (스위트스폿 존재 검증).
- 콤비네이션이 경기당 몇 회 성립하나, 그때 goal% 기여 (시그니처 무브 실효).
- 회귀: 기존 자기대국 goal% 밴드, career-sim, 사람 플레이 경로.

## 8. 단계 (빅뱅 금지 — 엔진 코어라 신중)

CLAUDE.md 불변식: 엔진 내부는 함부로 안 건드림. 이건 전술 코어 진화라 불가피하게
코어에 닿는다 → **옵트인 플래그 뒤에서**(defenseLoop 도입과 같은 패턴).

- **Phase 0**: 유인 캐리의 임계거리 도발 밴드 + 마커 커밋 롤 (기존 carry 확장,
  `baitCombo:false` 기본 옵트아웃). 직접 패스 막힘 명시.
- **Phase 1**: RECEIVER 드롭 커플링 — 유인 성립 시 마킹당하던 선수가 뒷공간으로
  이동(tx/ty), 오리엔테이션 FACING 예약. 3자 릴리스 도착 = linesBroken.
- **Phase 2**: 시각화(막힌 직접 레인 적색 + 열린 3자 길 녹색 + 리시버 드롭 화살표)
  + **hold 재정의**: 유인=캐리로 이관됐으므로 hold는 **"박자 늦추기"** 정체성
    (압박 리셋·재정비 — 게이지/모멘텀 관리, 상대 대형 재정렬 유도). 유인 도구가
    아님을 명확히. hold를 억지로 유인으로 부양하던 시도(4R~5R) 종결.
- **Phase 3**: 밸런스 — 콤비네이션 goal% 기여를 시그니처 수준으로(현 carry +1% →
  목표: 성립 시 확실한 전진), 남발 방지(임계거리·태클 리스크가 상한).

## 9. 열린 질문 (설계 결정 필요)

1. **릴리스를 몇 액션으로?** → **결정**: **자동**. 엔진이 가장 좋은 릴리스
   각의 3자를 잡고, 플레이어는 "릴리스"만 고른다(레이오프 대상 클릭 없음).
   턴제 조작 부담 최소화. 2액션 시퀀스(유인 캐리 → 릴리스) 유지.
2. **RECEIVER는 누구?** → **결정(2026-07-04)**: 대인(man/hybrid 마킹)이면
   **끌려나온 그 마커가 담당하던 우리 선수**가 드롭한다(마킹 관계 그대로 뒤집기 —
   가장 깨끗한 3자). 지역(zonal/midblock/lowblock)이면 **열린 뒷공간 최근접
   우리 선수**. → 함의: man 스킴에서 콤비가 가장 강력·정확(담당이 명확), 지역
   에선 느슨(누가 내려올지 상황 의존) — 스킴별 유인 감수성 차등이 자연 발생.
   구현 훅: man 계열은 마커→담당선수 배정을 읽고, 지역은 vacated 공간 최근접.
3. **드롭 타이밍**: → **결정**: **즉시**. 유인 성립 순간 리시버 드롭 armed
   (화면에 드롭 화살표), 다음 액션 릴리스로 완성. 유예 턴 없음(턴제 단순성).
4. **실패 시 비용**: 유인 캐리가 너무 가까워 뺏기면? → 기존 carry 태클 =
   턴오버 → 수비 국면(공수 왕복 재사용). 유인의 리스크가 실점 위험으로 번짐.
5. **스킴 의존**: 대인(man) 스킴에서 가장 강력(마커가 확실히 끌려옴), 지역
   (zonal/lowblock)에선 약함(안 끌려옴)? → 성향/스킴별 유인 감수성 차등이
   자연스러운 매치업 다양성.
