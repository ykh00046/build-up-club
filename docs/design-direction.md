# Build-Up Club 디자인 문서

## 1. 디자인 목표

이 문서는 Build-Up Club의 향후 설계 방향을 정리한다.

게임의 핵심 디자인 목표는 세 가지다.

1. 전술 판단이 실제 경기 결과에 영향을 준다.
2. 경기 결과는 설명 가능해야 한다.
3. 설명은 다음 성장 선택으로 이어져야 한다.

즉, 플레이어가 “왜 이겼는지/왜 졌는지”를 이해하고, 다음 경기에서 다른 결정을 시도하게 만드는 구조가 핵심이다.

## 2. 디자인 기둥

### 2.1 읽고 대응하는 축구

플레이어는 단순히 가장 낮은 위험도의 패스를 고르는 것이 아니라, 상대 구조를 읽고 대응해야 한다.

예시:

- 대인 압박은 같은 발밑 패스를 반복하면 위험하지만, 원투/써드맨으로 벗겨낼 수 있다.
- 지역 블록은 중앙 공간을 닫지만, 빠른 전환에는 흔들린다.
- 게겐프레스는 지연/운반을 응징하지만, 원터치 조합에 취약하다.

이 차이는 엔진 내부 수치뿐 아니라 UI 문구, 리포트, 훈련 선택까지 일관되게 드러나야 한다.

### 2.2 짧은 경기, 긴 성장

한 경기는 짧은 전술 모먼트다. 하지만 경기 후 선택은 장기 성장에 남는다.

장기 축:

- 포지션 레벨
- 디비전/승격
- 철학/퍽
- 클럽 정체성 XP
- 훈련 효과
- 시즌 목표

짧은 경기에서 생긴 사건이 장기 축으로 연결되어야 반복 플레이가 의미를 갖는다.

### 2.3 설명 가능한 결과

결과 리포트는 단순 요약이 아니다. 플레이어의 학습 장치다.

리포트가 답해야 하는 질문:

- 무엇이 잘 통했나?
- 무엇이 상대에게 읽혔나?
- 결정적 장면은 무엇이었나?
- 다음 경기에서 무엇을 바꿔야 하나?
- 어떤 훈련이 그 문제를 해결하나?

리포트는 훈련 선택과 직접 연결되어야 한다.

## 4. 핵심 데이터 모델

### 4.1 Club state

중요 persistent field:

- `saveVersion`
- `cash`
- `fans`
- `levels`
- `divIdx`
- `points`
- `matchday`
- `record`
- `effects`
- `philosophy`
- `perks`
- `philoPoints`
- `identityXp`
- `trainingEffects`

규칙:

- 새 persistent field는 `freshState()`와 `normalizeState()`에 반드시 추가한다.
- 저장 데이터는 `save()` 전에 정규화한다.
- `load()`는 구버전/깨진 데이터를 기본값으로 보정해야 한다.

### 4.2 TrainingEffect

훈련 효과는 경기 후 선택이 다음 경기로 넘어가는 연결 데이터다.

형태:

```js
{
  id: 'central_combo',
  label: '중앙 조합 훈련',
  tone: 'good',
  until: club.matchday + 1,
  actions: ['bounce', 'third_man'],
  multiplier: 0.88,
  score: {
    execAdd: 0.04,
    xgMul: 1,
    concedeMul: 1
  }
}
```

의미:

- `actions`: 다음 경기에서 위험도 보정을 받을 행동
- `multiplier`: 해당 행동의 tactical risk multiplier
- `score.execAdd`: 스코어라인 수행 점수 보정
- `score.xgMul`: xG 보정
- `score.concedeMul`: 실점 확률 보정
- `until`: 만료 matchday

기본 원칙:

- 대부분 1경기 지속
- 강한 수비 안정 효과만 2경기 가능
- 효과는 플레이어가 UI에서 읽을 수 있어야 한다.

### 4.3 Identity XP

현재 정체성:

- `positional`: 점유 조립형
- `direct`: 직선 전진형
- `wing`: 측면 전환형
- `pressproof`: 압박 회피형

획득 기준:

- 점유 조립형: 라인 통과, 원투, 써드맨
- 직선 전진형: 침투, 공간 패스, 골
- 측면 전환형: 전환, 풀백 전진
- 압박 회피형: 상황 해결, 경기 중 선택, 후방 안정

향후 확장:

- 정체성 레벨
- 정체성 퍽
- 정체성 기반 시즌 목표
- 정체성별 브리핑 추천

## 5. 상대 스카우팅 디자인

다음 핵심 기능은 상대 스카우팅 UI다.

### 5.1 목표

전술 브리핑에서 플레이어가 다음을 알 수 있어야 한다.

- 상대가 어떤 방식으로 압박/수비하는가
- 어떤 행동을 반복하면 위험한가
- 어떤 행동이 약점을 찌르는가
- 추천 라인 의도는 무엇인가
- 현재 활성 훈련 효과가 어떤 대응을 돕는가

### 5.2 Scheme별 디자인

| Scheme | 플레이어에게 보여줄 메시지 | 추천 행동 | 주의 행동 |
|---|---|---|---|
| `man` | 따라붙는 대인 압박. 같은 연결은 읽히지만 조합 플레이에 취약 | 원투, 써드맨 | 반복 발밑 패스 |
| `zonal` | 공간을 닫는 지역 블록. 중앙은 좁지만 반대 전환에 약함 | 전환, 약측 공략 | 무리한 중앙 침투 |
| `gegen` | 즉시 압박. 오래 끌면 위험하지만 원터치 우회가 통함 | 원투, 써드맨 | 기다리기, 운반 |
| `hybrid` | 선택적으로 덫을 놓는 압박. 커버 바깥의 세 번째 움직임이 중요 | 써드맨, 리셋 후 전진 | 정직한 전진 |

### 5.3 UI 구조

전술 브리핑에 “상대 스카우팅” 카드를 추가한다.

카드 구성:

- 상대 성향
- 약점
- 위험
- 추천 대응
- 활성 훈련 효과

예시:

```text
상대 스카우팅
대인 압박: 마크가 빠르게 따라붙습니다.
약점: 원투/써드맨으로 첫 압박을 벗기면 라인이 흔들립니다.
주의: 발밑 패스를 반복하면 추적당합니다.
추천: 전방 내려와 연결 + 중원 라인 사이
활성 훈련: 중앙 조합 훈련 — 원투/써드맨 위험도 감소
```

### 5.4 검증 기준

- scheme별 브리핑 문구가 표시된다.
- 문구의 추천 행동과 `tacticalFactors()`의 실제 보정이 일치한다.
- 활성 `trainingEffects`가 브리핑에 표시된다.
- 모바일에서 카드가 너무 길어지지 않는다.

## 6. 리포트/훈련 디자인

### 6.1 결과 리포트의 역할

리포트는 다음 선택을 위한 근거다.

리포트 항목:

- 잘 통함
- 읽힘
- 결정적 장면
- 다음 경기 추천

훈련 선택지는 리포트의 원인과 연결되어야 한다.

예시:

- “전환 반복이 읽힘” → 전환 패턴 훈련 / 중앙 조합 훈련
- “압박 상황 해결” → 압박 탈출 훈련 / 후방 균형 훈련
- “마무리 부족” → 침투 마무리 훈련 / 써드맨 반복 훈련

### 6.2 훈련 선택지 설계 원칙

각 선택지는 최소 두 가지 효과를 가져야 한다.

- 즉시 성장: 포지션 레벨 +1
- 장기 성장: 정체성 XP
- 단기 전술 효과: 다음 경기 행동/스코어 보정

선택지는 trade-off를 가져야 한다.

예:

- 중앙 조합 훈련: 다음 경기 조합 플레이 강화
- 전환 패턴 훈련: 다음 경기 전환 강화
- 후방 균형 훈련: 공격 효과는 약하지만 실점 억제

## 7. 밸런스 원칙

### 7.1 위험도 보정

훈련/상황/상대 scheme 보정은 곱연산으로 누적된다.

권장 범위:

- 약한 보정: `0.93~0.95`
- 표준 보정: `0.88~0.91`
- 강한 일회성 보정: `0.84~0.87`

하한은 `tacticalRiskMultiplier()`의 clamp로 통제한다.

### 7.2 스코어라인 보정

권장 범위:

- `execAdd`: `0.03~0.04`
- `xgMul`: `1.04~1.08`
- `concedeMul`: `0.90~0.95`

원칙:

- 액션 위험도 보정은 “경기 중 체감”을 만든다.
- 스코어라인 보정은 “결과 정산 체감”을 만든다.
- 둘 다 너무 강하면 성장 곡선이 무너진다.

### 7.3 정체성 보너스

정체성은 장기 방향성을 제공해야 하지만, 단일 정답이 되면 안 된다.

대응:

- 상대 scheme별 유불리를 다르게 둔다.
- 정체성별 퍽은 특정 상황에서만 강하게 만든다.
- 브리핑에서 “이번 상대에게 유리한 정체성”을 보여줄 수 있다.

## 8. 테스트 전략

현재 유지해야 할 테스트 축:

- `events-test`: 커리어 이벤트/미션/효과
- `variety-math`: 수식 불변식
- `career-sim`: 장기 커리어 무결성
- `engine-depth-test`: 전술 factor/상황 이벤트
- `tactical-report-test`: 리포트 품질
- `identity-training-test`: 정체성/훈련/마이그레이션
- `dom-check`: DOM 참조 무결성
- `smoke`: Chromium 통합 플로우

새 기능 추가 시 테스트 기준:

- 새 persistent field 추가 → migration test 필수
- 새 scheme 문구 추가 → 브리핑 표시 test 필수
- 새 tactical factor 추가 → factor id와 multiplier test 필수
- 새 훈련 효과 추가 → 저장/전달/만료 test 필수
- 새 UI 요소 추가 → DOM check 또는 smoke 보강
